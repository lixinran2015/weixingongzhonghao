Part 1：基于 Wiki 的 LLM 目录方法

用四个目录 + 一份规则把「原件、知识、临时产出」和「AI 怎么动手」拆开；下面是结构示意（与 Part 2 真实实现可一一对照）。

raw/

原始资料

飞书、网页、PDF

 →

wiki/

知识层

按主题整理

 →

outputs/

工作台

临时结论、简报
 

CLAUDE.md

告诉 AI 怎么组织这套系统的规则文件

一句话：原件落 raw/，可沉淀知识落 wiki/，草稿与简报落 outputs/；边界与流程写在 CLAUDE.md（亦可用 AGENTS.md 等同类文件）。

Part 2：系统架构（真实实现）

这是我们实际部署的架构，不是概念图，是跑在生产环境的代码结构。

关键设计：WikiPage 分两种类型——Source Wiki（is_source_wiki=true，如 source-xxx）是原始内容的镜像；Concept Wiki（is_source_wiki=false，如 Cursor、小龙虾）是提取的概念页。双链从 Source Wiki 指向 Concept Wiki，最后聚合成 KnowledgeGraph 缓存供前端可视化。

Part 3：六步工程闭环

以下是我们处理 5248 篇 Source 时沉淀的工程流程，与 Part 2 的架构对应，纯粹是解决具体问题后的结果。

六步顺时针闭环：分类隔离 → 幂等检查 → 受限提取 → 建立链接 → 同步缓存 → 清理噪声。若后台不显示 SVG，请截屏该图插入。

阶段 1：分类隔离

  • 按领域标签对文章分组（股票/装修/旅行/技术）
  • 每个领域单独提取实体，避免同名概念语义混乱
  • 股票里的「龙头」≠ 装修里的「龙头」

阶段 2：受限提取

  • 每篇最多提取 5 个核心概念
  • 排除：人名、地名、日期、泛词（市场/生活/问题等）
  • 避免一篇「离职感言」抽出 34 个人名的悲剧

阶段 3：建立链接

  • 创建 WikiLinks（Source Wiki → Concept Wiki）
  • 在内容中添加 [[双链]]：用 (?<!\[)name(?!\]) 匹配，排除已链接的
  • 按名称长度降序匹配，避免短词覆盖长词

阶段 4：同步缓存

  • 更新 KnowledgeGraph 缓存表：nodes、edges、统计、时间
  • 只更新业务表不同步缓存 → 前端图谱空白

Part 4：7 个坑（实战记录）

以下问题都是实际遇到、且造成过数据混乱的真实情况：

# 问题 后果
1

不分类直接抽实体

股票、装修、旅行混在一起处理

 同名概念语义混乱，知识图谱不可用
2

一篇抽出几十个实体

不加数量限制

 一篇「离职感言」抽出 34 个人名，热门列表被淹没
3

热门阈值固定

阈值设为 3，不区分数据量

 新领域（如装修）概念引用少，全部被过滤，看不到
4

数据库更新，前端不显示

只更新业务表，未同步缓存

 KnowledgeGraph 缓存表未更新，前端图谱空白
5

重复处理

多次运行脚本，不做幂等检查

 实体数从 6000 涨到 12925，一半是一次性噪声
6

中文正则匹配失效

用 \b 匹配中文

 完全失效；已加 [[链接]] 的被重复处理
7

数据只增不减

没有清理机制

 低频噪声堆积，热门概念被淹没

Part 6：基于这些数据能做什么

1. 主题溯源：追踪一个概念的真实演进

  • 「Cursor」：从第一次听说（哪篇文章）→ 安装尝试 → 日常使用 → 深度依赖的完整时间线
  • 「小龙虾」：从最初的概念 → 实际搭建 → 遇到问题的真实记录
  • 「上海天文馆」：想去 → 规划 → 实际去过的决策过程

2. 交易复盘：把零散记录变成可回顾的决策链

  • 某只股票（如「科陆」）的所有提及记录：什么时候第一次提到 → 买入理由 → 持仓过程 → 卖出/继续持有
  • 特定交易模式（如「首板模式」）的实践记录：理论 → 尝试 → 结果 → 调整
  • 发现重复犯的错误：哪些概念被反复提及但结果相似

3. 阅读清单：按主题聚合跨来源的内容

  • 「装修」主题：热河公众号文章 + 自己的装修决策记录 + 相关购物清单
  • 「读书」主题：成长类笔记中提到的书籍 → 实际阅读记录 → 后续实践
  • 「旅行」主题：所有地点相关的日记 + 攻略收藏 + 实际行程

核心逻辑：

不是「整理已有内容」，而是「发现隐性连接」→ 跨时间、跨领域、跨媒介的概念关联。

Part 7：ToB 场景 vs RAG

对比 传统 RAG LLM Wiki
检索方式 向量相似度，容易漏掉相关但表述不同的内容 概念图谱导航，找到「相关概念」的所有内容
回答质量 单次检索，容易遗漏上下文 先聚类主题，再按主题生成，结构完整
可解释性 黑盒，不知道用了哪些资料 白盒，通过 [[双链]] 可追溯来源
知识沉淀 用完即走，不产生资产 生成 Concept Wiki，持续积累企业知识资产
